# Simplify Patterns

Recurring code quality issues identified by /simplify across executions.
Read this before implementing to avoid repeating past mistakes.

## Patterns

- **Console.Out-capture test boilerplate duplicates silently**: any new test that needs to inspect Serilog's JSON console output (redirect `Console.Out` to a `StringWriter`, restore in `finally`, filter/parse lines) should use the `ConsoleLogCapture` helper (`CaptureAsync`/`ParseLogLinesWithTraceId`) rather than re-inlining the pattern. It grew to 3 near-identical copies within `EventLedger.Gateway.Tests` before being extracted during issue #4's `/simplify` pass — but a 4th copy in `EventLedger.AccountService.Tests` (`AccountServiceLoggingTests.cs`) was missed in that pass and only caught by the follow-up `/workflow-review`. `ConsoleLogCapture` is `internal`, so it exists as a **deliberate, separate copy per test project** (`tests/EventLedger.Gateway.Tests/ConsoleLogCapture.cs` and `tests/EventLedger.AccountService.Tests/ConsoleLogCapture.cs`) — same rationale as `SqliteTempDbFixture`, since this codebase's test projects don't reference each other. Within one test project, always reuse the local copy; don't inline the pattern again. *(First seen: 2026-07-15; Account Service copy added: 2026-07-16)*
- **A temp directory built solely to hold one SQLite file is unnecessary ceremony**: when a test needs to simulate "database unreachable" by deleting the underlying file after a real host has already started against it, deleting just the file (`File.Delete(dbPath)`) is sufficient — `SqliteConnection.ClearAllPools()` plus a missing file reliably fails the next open attempt, it does not get silently recreated. A wrapping `Directory.CreateDirectory`/`Directory.Delete(tempDir, recursive: true)` pair adds setup/teardown surface with no isolation benefit, since the unique temp file name already provides per-test isolation. (WAL-mode sidecar files — `-wal`/`-shm`, left over from `Database.EnsureCreated()`'s `PRAGMA journal_mode = 'wal'` — referencing a since-deleted main file are the likely reason SQLite fails to reopen here rather than silently recreating an empty database, unlike deleting a whole parent directory up front *before* startup, which crashes host boot instead — see `tests/EventLedger.Gateway.Tests/HealthControllerTests.cs`.) *(First seen: 2026-07-16)*
- **`MeterListener`-based metric test boilerplate**: any new test that needs to observe an `OpenTelemetry`/`System.Diagnostics.Metrics` `Counter`'s recorded values (subscribe a `MeterListener` to a named `Meter`, filter to matching instruments, capture `(value, tags)` per measurement into a list) should reuse the `CreateListener`/`RecordedMeasurement` shape already in `tests/EventLedger.Gateway.Tests/RequestMetricsMiddlewareTests.cs` and `tests/EventLedger.AccountService.Tests/RequestMetricsMiddlewareTests.cs` rather than re-inventing it. Like `ConsoleLogCapture`/`SqliteTempDbFixture`, this exists as a **deliberate, separate copy per test project** — confirmed via `/simplify`'s reuse-review pass to be the established, correct pattern here, not an oversight — since this codebase's test projects don't reference each other. Within one test project, reuse the local `CreateListener` helper for any additional `MeterListener`-based test. *(First seen: 2026-07-16)*
- **A configurable stub `HttpMessageHandler` subsumes a fixed-status one**: when a test project already has (or gains) a stateful stub that can be parameterized to "always return status X" (e.g. `FlakyAccountServiceHandler(failuresBeforeSuccess: 0)` always returns success), a separate fixed-status-only handler (`StubAccountServiceHandler`) becomes redundant — check every call site of the simpler handler's factory method before assuming both are still needed; if none pass a non-default value, the fixed handler and its dedicated factory overload can be deleted in favor of the configurable one. Found during issue #6's `/simplify` pass: `StubAccountServiceHandler`/`CreateFactory(HttpStatusCode)` were fully subsumed by the newly-added `FlakyAccountServiceHandler`/`CreateFactory(HttpMessageHandler)` — all 7 existing call sites used the default status. *(First seen: 2026-07-16)*
- **A stub `HttpMessageHandler` that returns a fixed status + body belongs in one shared file, not one copy per new test file**: unlike `ConsoleLogCapture`/`SqliteTempDbFixture` (deliberately duplicated per test *project* because they're `internal`/cross-project), a handler needed by two test *classes in the same project* has no such barrier — write it once as a standalone class (see `tests/EventLedger.Gateway.Tests/StubResponseHandler.cs`) and reference it from both. Found during issue #7's `/simplify` pass: `BalanceQueryHandlerTests.cs` and `AccountsControllerTests.cs` each introduced an identical `(HttpStatusCode status, string body) : HttpMessageHandler` class in the same commit — neither predates the other, so it was pure duplication, not an evolving-requirements accident. *(First seen: 2026-07-16)*
- **Reopening a factory/database to change Account Service behavior mid-test, when the code under test never calls the Account Service**: before spinning up a second `WebApplicationFactory` (or reopening the same SQLite file under a new one) just to switch the stubbed Account Service response between two phases of a test, check whether the endpoint under test actually depends on that handler at all — `EventQueryHandler` (backing `GET /events/{id}` and `GET /events?account=...`) never calls it, so a single `FlakyAccountServiceHandler` instance with its mutable `FailuresBeforeSuccess` property (already established for RES-4's half-open/close test) is enough to represent "Account Service reachable, then not" without a second factory. Found during issue #7's `/simplify` pass, in the RES-7 verification test. *(First seen: 2026-07-16)*
